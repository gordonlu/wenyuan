\# 项目原则



1\. 三席必须保持平等。

2\. 第一轮必须独立。

3\. 不传递隐藏推理，只交换结构化结果。

4\. 多数意见不能抹去少数意见。

5\. 多数不等于事实，只代表当前合议结果。

6\. 每次决策都应可追溯。

7\. 不为了多 Agent 而多 Agent。

8\. 先证明价值，再增加复杂基础设施。

9\. 执行者不能仅凭自己声明任务完成。

10\. 所有真实模型行为都必须可测试、可回放、可比较。

\# Phase 1：稳定核心流程



目标：把 MVP 从“能运行”提升到“可重复运行、可诊断、不会轻易卡死”。



\## 1.1 状态机加固



\* \[x] 所有阶段转换统一经过 `wenyuan-core`

\* \[x] 禁止重复启动同一阶段

\* \[x] 防止浏览器重复点击造成重复执行

\* \[x] 防止同一 Session 被两个请求同时推进

\* \[x] 每个阶段写入明确的 started / completed / failed 状态

\* \[x] 服务异常退出后能够识别未完成的 Session

\* \[x] 为运行中的阶段增加 lease 或 execution token

\* \[x] 明确恢复策略：



&#x20; \* 可继续

&#x20; \* 可重试

&#x20; \* 必须人工重启

&#x20; \* 不可恢复



\## 1.2 错误处理



\* \[x] Provider 超时有明确错误

\* \[x] 单席失败不会被误判为阶段完成

\* \[x] JSON 解析失败保存原始响应

\* \[x] 格式修复请求最多执行一次

\* \[x] 重试操作不会产生重复 Idea、Proposal 或 Vote

\* \[x] 取消后不再继续推进后续阶段

\* \[x] 前端显示可理解的错误原因

\* \[x] 日志中不输出 API Key



\## 1.3 测试补齐



重点补以下测试：



\* \[x] 重复启动阶段

\* \[x] 并发推进同一 Session

\* \[x] 单席超时

\* \[x] 单席返回非法 JSON

\* \[x] 格式修复成功

\* \[x] 格式修复失败

\* \[x] 中途取消

\* \[x] 服务重启后的 Session 恢复

\* \[x] Convergence 只执行一次

\* \[x] 事件顺序与数据库状态一致



\### Phase 1 验收标准



\* 连续运行 20 次 Mock 讨论无卡死

\* 任意阶段失败后可以明确恢复或重试

\* 不出现重复 Idea、Proposal、Vote

\* 前端刷新不导致后台任务重复执行

\# Phase 3：讨论质量提升



目标：避免三席输出趋同、互相附和或只做表面批评。



\## 3.1 独议质量



\* \[x] 第一轮完全隔离

\* \[x] 每席必须列出假设

\* \[x] 每席必须列出风险

\* \[x] 每席至少提出一个非主流方向

\* \[x] 自动检测重复 Idea

\* \[x] 相似 Idea 合并但保留来源席位

\* \[x] 限制每席最多 5 个 Idea



\## 3.2 批议质量



每次批议必须包含：



\* \[x] 对方最强的一点

\* \[x] 对方最弱的一点

\* \[x] 一个隐藏假设

\* \[x] 一个反例或失败条件

\* \[x] 一个具体改进建议

\* \[x] 一个需要补证的问题



禁止只输出：




\## 3.3 复议质量

\* \[x] 标记哪些观点被拒绝

\* \[x] 说明拒绝理由

\* \[x] 记录相较独议发生了哪些修改

\* \[x] 允许合并其他席位的 Idea

\* \[x] 允许撤回原判断

\* \[x] 更新置信度



\## 3.4 讨论质量指标



先实现简单指标：



\* \[x] Idea 重复率

\* \[x] 三席观点相似度

\* \[x] 批议有效率

\* \[x] 复议修改率

\* \[x] 自投率

\* \[x] 票数集中度

\* \[x] 少数意见保留率

\* \[x] 平均 token 消耗

\* \[x] 平均完成时间



暂时不要引入复杂向量数据库，可先使用关键词、文本归一化或轻量 embedding 接口。



\### Phase 3 验收标准



\* \[x] 三席不再只是换措辞重复同一答案

\* \[x] 最终报告可以展示“观点如何演化”

\# Phase 5：表决与合议机制完善



目标：让多数决不只是简单数票。



\## 5.1 投票策略



支持可配置策略：



\* 允许自投

\* 禁止自投

\* 普通多数

\* 加权评分

\* 风险否决

\* 全票通过

\* 有条件通过



普通议题默认：



```text

2 / 3 多数

```



高风险议题可配置：



```text

3 / 3 一致

```



或者：



```text

2 / 3 多数

且持正席未提出 blocking issue

```



\## 5.2 有效票规则



一票至少包含：



\* 选择

\* 评分

\* 理由

\* 置信度

\* 关键依据

\* 阻塞问题



结构不完整的票不得静默计入。



\## 5.3 合案复议



无多数时：



\* \[x] 提取共同部分

\* \[x] 提取核心分歧

\* \[x] 生成候选合案

\* \[x] 三席独立评价合案

\* \[x] 只允许再次表决一次

\* \[x] 再次无多数则输出“阁议未决”



\## 5.4 少数留议



最终结果必须保留：



\* 少数方选择

\* 少数方主要理由

\* 少数方认为何种条件下应重新评估

\* 是否包含重大风险提醒



\### Phase 5 验收标准



\* 多数结果可解释

\* 少数意见不会消失

\* 高风险议题不会仅因 2:1 自动执行

\* 无多数时不会无限循环

\# Phase 7：基准测试与价值验证



目标：验证三席架构是否优于单 Agent 自我反思。



\## 7.1 对照模式



增加两种运行模式：



\### Single Agent



```text

单 Agent

→ 初稿

→ 自我批评

→ 修订

```



\### Wenyuan



```text

三席独议

→ 批议

→ 复议

→ 阁议

```



\## 7.2 测试议题



准备固定议题集：



\* 产品方向

\* 系统架构

\* 技术选型

\* 隐私与安全

\* 功能优先级

\* 商业模式

\* 开源项目路线

\* 故障根因分析



每条议题固定输入和约束。



\## 7.3 比较指标



\* 方案数量

\* 方案差异度

\* 风险覆盖

\* 隐藏假设数量

\* 可执行步骤完整度

\* 最终结果一致性

\* 人工偏好

\* token 成本

\* 完成耗时

\* 失败率



\## 7.4 人工盲评



输出匿名化：



```text

Result A

Result B

```



避免评审者知道哪个来自单 Agent、哪个来自文渊阁。



\### Phase 7 验收标准



至少完成 20 条议题对照测试，并回答：



\* 三席是否产生更多有效差异

\* 批议是否真的改善方案

\* 多数决是否过度保守

\* 少数意见是否经常更有价值

\* 成本增加是否值得

\# Phase 8：后续扩展



仅在前面价值验证成立后再考虑。



\## 8.1 多模型席位



\* 谋远席使用探索型模型

\* 经世席使用综合型模型

\* 持正席使用严谨型模型



支持每席独立 Provider。



\## 8.2 可选书记官



书记官不参与投票，只负责：



\* 整理共识

\* 检查结构完整性

\* 汇总冲突

\* 生成最终报告



默认关闭。



\## 8.3 工具调用



逐步开放：



\* 网页搜索

\* 文件读取

\* 代码搜索

\* 日志分析

\* 数据查询



三个席位应有独立工具轨迹，但可共享证据池。



\## 8.4 长期记忆



只保存：



\* 稳定用户偏好

\* 已确认项目事实

\* 历史决策

\* 被验证的假设

\* 决策后实际结果



不要保存完整隐藏推理。



\## 8.5 桌面化



当 Web 版本稳定后：



\* 加入 Tauri

\* 复用现有 Vue UI

\* 复用 Rust Core

\* 支持本地托盘或快捷键

\* 支持导入本地文件作为议题背景
