# Wenyuan Product/UI Review and Next Coding Outline

更新时间：2026-06-20

## 总判断

现有功能对“需要做复杂判断、希望保留审议过程和来源证据”的用户已经有吸引力。三席合议、单 Agent 对照、书记官报告、证据池、工具轨迹、联网搜索、文档解析、代码搜索、可导出的审计报告，这些组合起来已经不是普通聊天壳。

但对第一次打开产品的用户，还不够“立刻想用”。当前价值更多藏在流程和结果里，启动页没有把核心承诺讲清楚，运行中状态不够有生命力，结果页也还没有把“为什么这个结论可信”压缩成一个可扫读的摘要。下一阶段不应继续横向加工具，应该先把已有能力包装成更有感知的产品体验。

## 功能吸引力 Review

### 已经有吸引力的部分

1. 多席审议是明确差异点
   - 谋远、经世、持正三个角色定位清晰，比普通“多 Agent”更容易被用户理解。
   - 当前有创意池、批议、复议、投票、少数意见、质量指标，已经能体现“不是一次性回答”。

2. 外部事实来源开始成立
   - 搜索默认有 Bing / DuckDuckGo / Wikipedia fallback。
   - 文档解析支持 txt/md/json/log/csv/tsv/xlsx/xls/ods/pdf/docx。
   - 代码搜索能把当前项目事实作为 evidence。
   - 搜索、文档、代码都进入不可信来源安全层，有 hash、source_kind、trust_level、safety_flags。

3. 审计导出是高价值工作流
   - Markdown 分 brief/standard/audit。
   - 工作区和报告视图都能看到证据、工具轨迹、投票和运行统计。
   - 这对产品决策、代码方案评审、文档研究类用户是实用卖点。

4. 用户偏好配置方向正确
   - 只保存本地 JSON 默认配置，避免把 8.4 做成过重的长期记忆系统。
   - 默认模式、搜索开关、书记官、投票策略、模型、代码搜索根和文件大小上限都能被复用。

### 当前不足以打动用户的部分

1. 缺少“第一次就看懂价值”的入口
   - 新建议题页字段清楚，但还像一个表单，不像一个能让用户马上理解产品价值的决策工作台。
   - 缺少一两个高质量场景入口，例如“产品路线判断”“代码方案评审”“文档事实核验”。
   - 现在的示例文本不错，但没有转化成可点击的一键填充模板。

2. 缺少运行中的可感知过程
   - 三张席位卡已经有 hover 光效和角色感，但“思考中”时没有专属动效。
   - 当前执行中主要靠 phase、notice 和事件轮询，用户不会强烈感到三席正在工作。
   - 工具调用发生时没有明显的 live 状态，例如“正在搜索网络”“正在解析文件”“正在交叉批议”。

3. 结果可信度还需要更强的摘要层
   - 证据和工具轨迹已经记录，但用户需要先读很多卡片才知道“结论为何可信”。
   - 缺一个顶部的“可信度摘要”：来源数量、外部来源类型、未验证点、少数意见、风险阻塞。
   - 有 evidence flags，但 UI 上还没有形成可扫读的安全/可信分级。

4. 工具入口仍偏工程化
   - 文档解析和代码搜索功能已经可用，但入口文案和视觉层级仍像附加面板。
   - 代码搜索适合开发者，但非开发者会不理解它和议题结果的关系。
   - 数据查询、日志分析还只是解析基础，暂时不要继续扩成复杂功能，先把“来源证据”体验做扎实。

## UI Review

### 已经做得好的部分

1. 产品识别度比普通 dashboard 强
   - 侧栏、WENYUAN COUNCIL、三席角色卡、色彩分工已经建立了品牌感。
   - Microsoft YaHei 字体策略符合当前中文产品方向。

2. 工作台和报告视图分离是正确结构
   - 工作台适合操作、调试和追踪。
   - 报告适合阅读、导出和交付。

3. 角色卡是当前最有潜力的视觉资产
   - 谋远/经世/持正的颜色和卡面区分明显。
   - hover 倾斜、光效、角色说明让产品不像普通表格系统。

4. 基础可访问性有意识
   - 按钮使用 lucide 图标。
   - focus-visible、prefers-reduced-motion 已有基础。

### 明显欠缺

1. 席位运行态动画缺失
   - active/running 状态应有呼吸光、边框流动、状态脉冲或局部 shimmer。
   - 动效只放在正在工作的席位卡，不要让所有卡同时动。
   - 必须支持 `@media (prefers-reduced-motion: reduce)`。

2. 工具执行态缺失
   - 文档解析按钮只有“解析中”，但没有进度占位或 skeleton。
   - 代码搜索只有按钮文字变化，搜索结果区域没有 loading/skeleton。
   - 会话运行中没有显示“当前工具步骤”的紧凑状态条。

3. 新建议题页信息密度偏高
   - 外部资料、代码搜索、合议设置都直接展开，会让新用户觉得复杂。
   - 建议把“外部资料/代码搜索”改成可折叠来源区，并提供已添加来源数量和安全状态摘要。
   - 合议设置里高级项默认收起，保留模式、搜索、书记官三个核心开关。

4. 结果页缺少层级压缩
   - 证据、工具轨迹、创意池、批议、差异、提案、投票、质量、报告都存在，但用户容易失去主线。
   - 顶部应先显示“最终判断 / 关键依据 / 主要风险 / 下一步”四块。
   - 证据和工具轨迹可默认摘要，展开后才显示完整审计信息。

5. 空状态和错误状态不够产品化
   - 首次进入历史页、没有证据、没有工具轨迹、搜索无结果时，需要更明确的空状态。
   - 错误状态需要给出下一步，例如“调整代码搜索根目录”“降低文件大小”“检查搜索 provider”。

## 下阶段编码大纲

### P0：让“正在审议”可见

目标：用户点击“创建并开议”后，第一眼能感到三席正在工作，而不是页面静止等待。

改动范围：
1. `web/src/components/SeatRoleCard.vue`
   - 增加 `isActive` / `isRunning` 计算状态。
   - active 卡片增加呼吸边框、微弱光带、状态点脉冲。
   - failed 卡片保留重试按钮但视觉上与 active 区分。
   - reduced motion 下关闭循环动画，只保留静态高亮。

2. `web/src/components/PhaseProgressBar.vue`
   - 当前 phase 增加短动效，不要整条进度条同时动。
   - 增加一句紧凑状态文案，例如“谋远席正在独议”“三席正在交叉批议”。

3. `web/src/views/SessionWorkspaceView.vue`
   - 顶部 notice 改成运行状态条，显示当前 phase、运行席位、最近事件时间。
   - 事件轮询更新时避免页面跳动。

验收标准：
1. 运行中的席位卡有明确动态反馈。
2. 三席同时运行时最多 1-2 个主要动效区域。
3. `prefers-reduced-motion` 下没有循环动画。
4. `pnpm run typecheck` 通过。

### P0：补结果可信度摘要

目标：让用户在 10 秒内知道结论是否可信、依据来自哪里、还有什么风险。

改动范围：
1. 新增 `web/src/components/EvidenceSummary.vue`
   - 展示外部来源数量、文件/代码/网页来源分布、疑似注入数量、未验证 claim 数量。
   - 对 `untrusted_external` 用明确但不恐吓的提示。

2. 新增 `web/src/components/DecisionDigest.vue` 或扩展 `DecisionSummary.vue`
   - 顶部展示：最终判断、关键依据、主要风险、少数意见、下一步。
   - 工作台和报告视图都复用。

3. `web/src/domain/session.ts`
   - 增加纯前端聚合函数：`evidenceSummary(details)`、`decisionDigest(details)`。
   - 用单元测试覆盖空 evidence、只有内部 evidence、外部来源含安全 flags 三种场景。

验收标准：
1. 报告顶部不用展开审计区也能看懂结果可信度。
2. 安全标记在摘要层可见。
3. `pnpm run typecheck` 和相关单元测试通过。

### P1：把新建议题页从表单变成启动工作台

目标：降低新用户启动成本，让用户知道该把什么材料交给文渊。

改动范围：
1. `web/src/views/NewSessionView.vue`
   - 增加三类一键模板：产品决策、代码方案评审、文档事实核验。
   - 点击模板后填充标题、议题、背景，并建议是否启用搜索/书记官。
   - 外部资料和代码搜索改为“来源区”，默认折叠或半折叠。

2. `web/src/components/DocumentSourcePanel.vue`
   - 增加空状态：说明支持格式、大小上限、注入安全处理。
   - 解析中显示 skeleton 文件卡。

3. `web/src/components/CodeSearchPanel.vue`
   - 增加搜索根目录提示和无结果状态。
   - loading 时显示 skeleton 匹配行。

验收标准：
1. 新用户可以一键填入高质量示例并开议。
2. 不上传文件、不搜索代码时页面仍清爽。
3. 有来源时能看到来源数量和安全状态。

### P1：让工具轨迹更像“可验证来源”，不是日志列表

目标：把工具轨迹从工程记录升级成用户可理解的证据链。

改动范围：
1. `web/src/views/SessionWorkspaceView.vue`
   - 工具轨迹默认显示摘要：搜索了什么、读了哪些文件、找到多少代码命中。
   - 详细输入 hash、耗时、error 放进展开区。

2. `web/src/domain/session.ts`
   - 增加 `toolRunSummary()` 和 `toolRunStatusLabel()`。

3. 后端保持当前 `tool_runs` 结构，不急着改数据库。

验收标准：
1. 非工程用户也能理解工具为结论提供了什么。
2. Audit 导出仍保留完整轨迹。

### P1：补产品化空状态和错误修复路径

目标：错误不只是报错，而是告诉用户下一步怎么处理。

改动范围：
1. `web/src/components/ApiErrorState.vue`
   - 支持可选 `hint` 和 `action`。

2. 文档解析错误
   - 文件过大：提示偏好页可调 `max_file_size_mb`。
   - 不支持格式：提示当前支持格式。
   - 注入风险：提示“仍可作为材料，但不会被当作指令执行”。

3. 代码搜索错误
   - 根目录非法：提示去配置页修改代码搜索根。
   - 查询为空/无结果：给示例查询。

验收标准：
1. 常见错误有明确下一步。
2. 错误提示不暴露内部栈信息。

### P2：报告视图打磨

目标：让报告可以直接作为交付物，而不是工作台的另一种排列。

改动范围：
1. 报告顶部增加封面式摘要，但不要做营销 hero。
2. 把审计信息默认移到报告下半部分。
3. Markdown/HTML 导出同步新摘要结构。
4. HTML 导出补证据和工具轨迹摘要。

验收标准：
1. 导出的 HTML 不需要打开应用也能解释结论。
2. brief/standard/audit 三档内容层级清楚。

### P2：暂缓事项

1. 8.5 Tauri 继续延后，等 Web 体验稳定。
2. 不做复杂长期记忆，继续只保留 JSON 用户偏好。
3. 数据查询先不要上 SQL 或聚合 UI，当前只保留表格解析和证据化。
4. 日志分析可以下一轮做轻量 level/时间/错误聚类，但不要先做复杂可视化。

## 推荐执行顺序

1. P0 运行态动画与状态条。
2. P0 可信度摘要和结果 Digest。
3. P1 新建议题模板与来源区收敛。
4. P1 工具轨迹摘要化。
5. P1 错误/空状态产品化。
6. P2 报告导出打磨。

完成前两项后，现有功能会从“能力已经有”变成“用户能明显感知到它在帮我做判断”。这比继续堆更多工具更重要。
